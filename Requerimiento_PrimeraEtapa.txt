# Proyecto Quini 6 – Etapa 1

## Obtención y visualización del histórico

## 1. Objetivo

Desarrollar la primera etapa de una aplicación web que permita obtener, almacenar y consultar el histórico de resultados del Quini 6 para las modalidades:

* Tradicional / Primer Sorteo
* La Segunda del Quini

En esta etapa no se realizarán predicciones, estadísticas avanzadas ni entrenamiento de modelos de Machine Learning.

El alcance se limita a:

1. Obtener los resultados históricos mediante scraping.
2. Guardar los resultados en una base de datos MySQL.
3. Actualizar únicamente los sorteos que todavía no estén guardados.
4. Visualizar y verificar la información descargada desde una interfaz web.

# 4. Pantalla “Histórico de jugadas”

La pantalla debe incluir:

## Encabezado

Título:

**Histórico de jugadas del Quini 6**

Descripción breve:

“Consulta y actualización de los resultados históricos del Primer Sorteo y La Segunda.”

## Información de actualización

Mostrar:

* Fecha y hora de la última ejecución del scraper.
* Último número de sorteo guardado.
* Fecha del último sorteo guardado.
* Cantidad total de sorteos almacenados.
* Cantidad total de jugadas almacenadas.
* Estado de la última actualización:

  * Exitosa
  * Exitosa con advertencias
  * Fallida
  * En proceso

## Botón de actualización

Agregar un botón:

**Bajar históricos**

Al presionarlo, el sistema debe:

1. Consultar en MySQL cuál es el último sorteo almacenado.
2. Determinar qué sorteos faltan descargar.
3. Descargar únicamente los sorteos faltantes.
4. Extraer los resultados de:

   * Primer Sorteo
   * La Segunda
5. Validar los datos.
6. Guardarlos en la base de datos.
7. Mostrar un resumen de la operación.

El resumen debe indicar:

* Sorteos encontrados.
* Sorteos nuevos descargados.
* Jugadas insertadas.
* Sorteos ya existentes omitidos.
* Errores encontrados.
* Tiempo total del proceso.

Mientras se ejecuta el proceso:

* Deshabilitar el botón.
* Mostrar un indicador de carga.
* Mostrar el texto “Actualizando históricos...”.
* Evitar que el usuario pueda iniciar dos actualizaciones simultáneas.

---

# 5. Comportamiento incremental

El scraper no debe volver a descargar todo el histórico cada vez que se ejecuta.

Debe consultar:

* Último número de sorteo almacenado.
* Fecha del último sorteo almacenado.

A partir de esa información debe intentar obtener solamente los sorteos posteriores.

La prevención de duplicados no debe depender únicamente de la lógica del código. También debe existir una restricción única en MySQL.

La combinación que identifica una jugada será:

* Número de sorteo.
* Modalidad.

Por ejemplo:

* Sorteo 3392 + Tradicional
* Sorteo 3392 + Segunda

No puede existir más de un registro para la misma combinación de sorteo y modalidad.

---

# 6. Primera carga

Cuando la base de datos no tenga resultados, el sistema debe realizar una carga inicial completa.

La aplicación debe permitir configurar desde qué año o desde qué sorteo comenzar la importación.

Variables sugeridas:

```env
SCRAPER_START_YEAR=1988
SCRAPER_START_DRAW=
SCRAPER_BASE_URL=
```

Si `SCRAPER_START_DRAW` tiene un valor, debe tener prioridad sobre el año inicial.

La URL del sitio utilizado para el scraping debe estar configurada en variables de entorno y no escrita directamente en distintos archivos del código.

```text
TRADICIONAL
SEGUNDA
```

En la interfaz deberán mostrarse como:

* Primer Sorteo
* La Segunda

No importar todavía:

* Revancha

# 8. Datos que deben guardarse

Por cada modalidad se deben guardar:

* Número de sorteo.
* Fecha del sorteo.
* Modalidad.
* Número 1.
* Número 2.
* Número 3.
* Número 4.
* Número 5.
* Número 6.
* URL de origen.
* Fecha y hora de descarga.
* Fecha y hora de creación del registro.
* Fecha y hora de última modificación.

Los seis números deben guardarse ordenados de menor a mayor.

Los valores válidos son números enteros entre:

```text
0 y 45
```

No deben existir números repetidos dentro de la misma jugada.

# 9. Modelo de base de datos

## Tabla: `quini_sorteos`

Representa la información general de cada sorteo.

Campos sugeridos:

```sql
CREATE TABLE quini_sorteos (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    numero_sorteo INT UNSIGNED NOT NULL,
    fecha_sorteo DATE NOT NULL,
    url_origen VARCHAR(500) NULL,
    fecha_descarga DATETIME NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
        ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_quini_sorteo_numero (numero_sorteo),
    INDEX idx_quini_sorteo_fecha (fecha_sorteo)
);
```

## Tabla: `quini_jugadas`

Representa cada modalidad dentro de un sorteo.

```sql
CREATE TABLE quini_jugadas (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    sorteo_id BIGINT UNSIGNED NOT NULL,
    modalidad ENUM('TRADICIONAL', 'SEGUNDA') NOT NULL,
    numero_1 TINYINT UNSIGNED NOT NULL,
    numero_2 TINYINT UNSIGNED NOT NULL,
    numero_3 TINYINT UNSIGNED NOT NULL,
    numero_4 TINYINT UNSIGNED NOT NULL,
    numero_5 TINYINT UNSIGNED NOT NULL,
    numero_6 TINYINT UNSIGNED NOT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
        ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_quini_jugada_sorteo_modalidad (
        sorteo_id,
        modalidad
    ),
    CONSTRAINT fk_quini_jugada_sorteo
        FOREIGN KEY (sorteo_id)
        REFERENCES quini_sorteos(id)
        ON DELETE CASCADE
);
```

## Tabla: `quini_importaciones`

Debe registrarse cada ejecución del scraper.

```sql
CREATE TABLE quini_importaciones (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    fecha_inicio DATETIME NOT NULL,
    fecha_fin DATETIME NULL,
    estado ENUM(
        'EN_PROCESO',
        'EXITOSA',
        'ADVERTENCIA',
        'FALLIDA'
    ) NOT NULL,
    sorteo_inicial INT UNSIGNED NULL,
    sorteo_final INT UNSIGNED NULL,
    sorteos_procesados INT UNSIGNED NOT NULL DEFAULT 0,
    sorteos_insertados INT UNSIGNED NOT NULL DEFAULT 0,
    jugadas_insertadas INT UNSIGNED NOT NULL DEFAULT 0,
    registros_omitidos INT UNSIGNED NOT NULL DEFAULT 0,
    cantidad_errores INT UNSIGNED NOT NULL DEFAULT 0,
    detalle TEXT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

## Tabla opcional: `quini_importacion_errores`

Para registrar errores individuales sin detener toda la importación.

Campos sugeridos:

* ID.
* ID de importación.
* Número de sorteo.
* URL.
* Tipo de error.
* Descripción.
* HTML o fragmento recibido, limitado a un tamaño razonable.
* Fecha y hora.

---

# 10. Transacciones

Cada sorteo debe guardarse utilizando una transacción MySQL.

Proceso:

1. Insertar o actualizar `quini_sorteos`.
2. Insertar la jugada Tradicional.
3. Insertar la jugada Segunda.
4. Confirmar la transacción.

Si una de las dos jugadas es inválida o falta, no guardar el sorteo como completo.

En ese caso:

* Hacer rollback.
* Registrar el error.
* Continuar con el siguiente sorteo.

---

# 11. Validaciones del scraper

Antes de guardar un sorteo se debe validar:

* Número de sorteo válido.
* Fecha válida.
* Existencia de Tradicional.
* Existencia de La Segunda.
* Exactamente seis números por modalidad.
* Números enteros.
* Números entre 0 y 45.
* Sin números repetidos.
* Números ordenados de menor a mayor.
* Tradicional y Segunda identificadas correctamente.
* Sorteo no duplicado.

Si el HTML cambió y no se puede identificar la información, no guardar datos parciales como si fueran correctos.

---

# 12. Manejo de errores

El scraper debe ser tolerante a fallos.

Un error en un sorteo no debe cancelar necesariamente toda la importación.

Debe contemplar:

* Página no encontrada.
* Timeout.
* Error de conexión.
* HTML inesperado.
* Resultado incompleto.
* Números inválidos.
* Fecha inválida.
* Sorteo ya existente.
* Error de MySQL.
* Bloqueo temporal del sitio.
* Respuesta HTTP 403, 429 o 500.

Agregar:

* Reintentos automáticos configurables.
* Pausa entre solicitudes.
* Timeout configurable.
* User-Agent identificable.
* Límite máximo de reintentos.

Variables sugeridas:

```env
SCRAPER_REQUEST_DELAY_MS=1000
SCRAPER_TIMEOUT_MS=15000
SCRAPER_MAX_RETRIES=3
```

No realizar solicitudes masivas sin pausa.

# 14. Agrupación visual

Los registros deben poder visualizarse agrupados jerárquicamente por:

```text
Año
  └── Mes
       └── Día / Sorteo
            ├── Primer Sorteo
            └── La Segunda
```

Ejemplo:

```text

           └── La Segunda:   08 21 23 33 36 41
```

La jerarquía debe poder expandirse y contraerse.

Además de esta vista agrupada, puede existir una vista tabular plana.

Agregar un selector:

* Vista agrupada.
* Vista tabla.

# 16. API del backend

## Consultar histórico

```http
GET /api/historico
```

Parámetros posibles:

```text
page
pageSize
year
month
modalidad
numeroSorteo
fechaDesde
fechaHasta
search
orderBy
orderDirection
```

Respuesta sugerida:

```json
{
  "success": true,
  "data": [],
  "pagination": {
    "page": 1,
    "pageSize": 25,
    "totalRecords": 0,
    "totalPages": 0
  }
}
```

## Consultar resumen

```http
GET /api/historico/resumen
```

Debe devolver:

* Último sorteo guardado.
* Fecha del último sorteo.
* Total de sorteos.
* Total de jugadas.
* Última importación.
* Estado de la última importación.

## Ejecutar actualización

```http
POST /api/historico/actualizar
```

Debe iniciar el proceso de scraping incremental.

No debe aceptar dos ejecuciones simultáneas.

## Consultar estado de actualización

```http
GET /api/historico/actualizacion/estado
```

Debe permitir que el frontend consulte el avance.

Respuesta sugerida:

```json
{
  "success": true,
  "status": "EN_PROCESO",
  "progress": {
    "processed": 25,
    "total": 100,
    "inserted": 24,
    "errors": 1
  }
}
```

## Consultar importaciones

```http
GET /api/importaciones
```

Debe mostrar el historial de ejecuciones del scraper.

## Consultar errores de una importación

```http
GET /api/importaciones/:id/errores
```

# 17. Control de ejecución simultánea

Debe existir una protección para impedir que se ejecuten dos importaciones al mismo tiempo.

Puede implementarse mediante:

* Estado `EN_PROCESO` en la tabla de importaciones.
* Lock en MySQL.
* Lock interno en el proceso Node.js.

La solución debe seguir funcionando correctamente aunque el usuario presione varias veces el botón.

Al iniciar la aplicación, si existe una importación marcada como `EN_PROCESO` desde hace demasiado tiempo, debe considerarse interrumpida y registrarse como fallida o pendiente de revisión.

Aunque no se desarrollen ahora, la estructura debe permitir incorporar posteriormente:

* Nuevas modalidades.
